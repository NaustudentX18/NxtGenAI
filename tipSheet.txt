  SECTION 5 — CONFIG TUNING REFERENCE
================================================================================

  Edit with: nano /home/pi/NxtGenAI/config.ini
  Restart after changes: sudo systemctl restart pentestgpt

  ── [paths] ────────────────────────────────────────────────────

  model       = /home/pi/models/tinyllama-1.1b-q4_0.gguf
    ↳ Swap for Phi-2-2.7B-Q4_0 for better reasoning (uses ~400 MB more RAM)
    ↳ Any GGUF model compatible with llama-cpp-python works here

  wordlists   = /home/pi/wordlists/
  rockyou     = /home/pi/wordlists/rockyou.txt
    ↳ Replace with custom wordlists for targeted engagements

  common10k   = /home/pi/wordlists/10k-common.txt
    ↳ Used for fast hydra brute (10k passes in <2 min on SSH)

  reports     = /home/pi/reports/
    ↳ Change to a USB mount point to auto-save reports externally

  usb_mount   = /media/pi/
    ↳ USB drives appear here when plugged in; Button C exports to first found

  ── [llm] ──────────────────────────────────────────────────────

  n_ctx        = 512
    ↳ Context window tokens. Increase to 1024 if you have swap headroom.
    ↳ Higher = better reasoning but more RAM. 512 is safe on Zero 2 W.

  n_threads    = 3
    ↳ Pi Zero 2 W has 4 cores. Use 3 to leave one for UI responsiveness.
    ↳ Set to 4 for maximum inference speed (OLED may lag slightly).

  max_tokens   = 200
    ↳ Max tokens per LLM response. 200 keeps responses concise.
    ↳ Increase to 400 for more detailed AI reasoning.

  temperature  = 0.3
    ↳ Lower = more deterministic (good for pentest steps).
    ↳ 0.0 = fully deterministic. 1.0 = creative/random.

  repeat_penalty = 1.1
    ↳ Prevents the LLM from looping on the same phrase.

  ── [oled] ─────────────────────────────────────────────────────

  i2c_address = 0x3C
    ↳ Most SSD1306 OLEDs are 0x3C. Some are 0x3D. Run i2cdetect to confirm.

  brightness_normal = 200
    ↳ 0–255. 200 is bright. Reduce for battery saving.

  brightness_dim = 80
    ↳ Applied automatically when battery drops below threshold_warn.

  refresh_rate = 0.1
    ↳ OLED updates every 100 ms (10 fps). Lower = smoother but more CPU.

  ── [power] ────────────────────────────────────────────────────

  threshold_warn  = 20
    ↳ Voice warning + OLED dim kicks in at this battery %.

  threshold_sleep = 10
    ↳ Saves current report and initiates safe shutdown at this %.

  poll_interval = 30
    ↳ Reads battery every 30 seconds. Reduce to 10 for faster response.

  ── [security] ─────────────────────────────────────────────────

  max_risk_score = 10
    ↳ Actions above this score are auto-skipped in Full Auto mode.
    ↳ Set to 10 to allow all actions (recommended for professional use).
    ↳ Set to 5 to only allow passive/read-only actions.

  confirm_threshold = 10
    ↳ Actions at or above this score require voice confirmation.
    ↳ Set to 10 to disable all confirmation prompts.

  ── [audio] ────────────────────────────────────────────────────

  vosk_model  = /home/pi/models/vosk-model-small-en-us
    ↳ Swap for vosk-model-en-us-0.22 (~1.8 GB) for significantly
      better recognition accuracy on noisy networks or thick accents.

  rate        = 16000
    ↳ Must match Vosk model (16000 Hz for all small/medium models).

  channels    = 1
    ↳ Mono mic input. Stereo inputs are auto-mixed to mono by PyAudio.

  ── [wifi] ─────────────────────────────────────────────────────

  interface = wlan0
    ↳ Built-in WiFi. For monitor mode, plug in a USB adapter and change
      this to wlan1. Recommended adapters: Alfa AWUS036ACH, TP-Link TL-WN722N.

  capture_timeout = 60
    ↳ How long airodump-ng runs before attempting crack.
    ↳ Increase to 120+ if the target network is quiet.

  ── [voice] ────────────────────────────────────────────────────

  voice     = en-gb
    ↳ espeak-ng voice. Options: en-us, en-gb, en-au, en-in, de, fr, es
    ↳ Full list: espeak-ng --voices

  speed     = 130
    ↳ Words per minute. Lower = clearer on small speakers. Range: 80–400.

  amplitude = 100
    ↳ Volume. 0–200. 100 is default. 200 is maximum.


================================================================================
  SECTION 6 — PENTEST MODES — WHAT THEY DO
================================================================================

  ── MODE 1: NETWORK RECON ──────────────────────────────────────

  Voice command: "network scan" or "recon"
  Risk score: 2–3 (passive, read-only)

  Step-by-step:
    1. nmap -sn (ping sweep) — discovers all live hosts on /24 subnet
    2. nmap -sS -T3 --open — SYN scan on discovered hosts
       Ports scanned: 22 (SSH), 80 (HTTP), 443 (HTTPS), 8080, 3306 (MySQL)

  Output includes:
    • IP addresses of all live hosts
    • Open ports per host
    • Service banner information
    • Estimated OS fingerprint

  Best for:
    • First step in any engagement — always start here
    • Mapping an unknown network before targeted attacks
    • Identifying exposed services on client's internal network
    • Verifying firewall rules are working as expected

  ── MODE 2: WIFI CRACK ──────────────────────────────────────────

  Voice command: "wifi crack" or "wifi"
  Risk score: 5–7 (active, requires monitor mode)

  Step-by-step:
    1. airmon-ng start wlan0      — enables monitor mode on wlan0mon
    2. airodump-ng                — scans for nearby access points (60 s)
    3. aireplay-ng -0             — sends 1 deauth frame to force handshake
    4. aircrack-ng -w rockyou.txt — dictionary attack on captured .cap file
    5. airmon-ng stop             — restores managed mode

  Output includes:
    • List of detected SSIDs and BSSIDs
    • Signal strength per AP
    • Captured .cap file (stored in /tmp during run)
    • Cracked password (if in wordlist)

  Best for:
    • Testing WPA2 PSK strength on client WiFi
    • Demonstrating why weak passwords are dangerous
    • Wireless security assessments
    • Red team engagements with physical access on-site

  NOTE: Built-in Pi wlan0 does NOT support monitor mode.
        Use a USB WiFi adapter (Alfa AWUS036ACH recommended).
        Change config.ini [wifi] interface = wlan1 after plugging in.

  ── MODE 3: WEB PENTEST ─────────────────────────────────────────

  Voice command: "web pentest" or "web"
  Risk score: 2–5 (active HTTP requests to target)

  Step-by-step:
    1. curl -I target              — grabs HTTP headers for analysis
    2. sqlmap --batch --smart      — SQL injection detection (level 1, risk 1)

  curl header analysis checks for:
    • Server version disclosure (Apache/nginx version in Server: header)
    • Missing X-Frame-Options (clickjacking vulnerability)
    • Missing Content-Security-Policy
    • Missing X-XSS-Protection
    • Missing Strict-Transport-Security (HSTS)
    • HTTP vs HTTPS usage

  sqlmap checks for:
    • Boolean-based blind SQL injection
    • Time-based blind SQL injection
    • Error-based SQL injection
    • UNION query-based injection
    • Stacked queries

  Output includes:
    • Full HTTP response headers
    • List of detected SQL injection points
    • Database type and version (if injectable)
    • sqlmap session files in /home/pi/reports/

  Best for:
    • Quick web app security health check
    • Demonstrating SQL injection risk to developers
    • Pre-engagement recon on client web properties
    • Testing after a developer claims they "fixed" SQLi

  ── MODE 4: FULL AUTO (AI ORCHESTRATED) ─────────────────────────

  Voice command: "full auto" or "auto" or "ai"
  Risk score: varies (AI decides next action based on findings)

  The AI ReAct loop runs up to 10 iterations:
    Thought  → AI reasons about what to do next
    Action   → AI calls a tool (nmap, sqlmap, hydra, etc.)
    Observation → Tool output fed back to AI
    Repeat   → AI decides next step based on observation

  Tools available to the AI:
    • nmap_scan     — network host discovery + port scan
    • wifi_crack    — WPA2 handshake capture + crack
    • web_pentest   — HTTP headers + SQL injection
    • arp_spoof     — ARP cache poisoning (MitM position)
    • sqlmap        — direct SQL injection testing
    • hydra_brute   — dictionary brute force on any service
    • report_done   — signals AI has finished, generates report

  AI behaviour:
    • Scores each action 1–10 before executing
    • Skips actions above max_risk_score threshold
    • Adapts strategy based on what each tool discovers
    • Automatically generates a final report

  Best for:
    • Hands-free automated assessments
    • Demonstrating AI-driven attack chaining to clients
    • Running overnight assessments while you sleep
    • Full engagement coverage with minimal operator input

  ── SCENARIO 1: SMALL BUSINESS INTERNAL NETWORK AUDIT ──────────

  Situation:
    Client runs a small accountancy firm. They want to know if their internal
    network is secure before applying for cyber insurance.

  You walk in with the Pi in your pocket. It's running off PiSugar 3.
  You plug a USB WiFi adapter into the Pi and SSH in over the LAN.

  Steps:
    1. Voice: "network scan" — maps all hosts on their LAN
    2. Review OLED results — note any unexpected open ports
    3. Voice: "web pentest" — target any internal web admin panels found
    4. Voice: "wifi crack" — test their office WiFi password strength
    5. Export report to USB: hold Button C, say "export"
    6. Present JSON + HTML report to client — shows exactly what's exposed

  Value delivered:
    • Professional HTML report with dark Bootstrap 5 theme
    • Risk-scored findings (1–10) for each issue found
    • Evidence of SQL injection / weak WiFi password / open ports
    • All generated in under 30 minutes, on-site, no laptop needed

  ── SCENARIO 2: HOTEL / CONFERENCE WIFI SECURITY TEST ───────────

  Situation:
    A hotel chain hires you to test the security of their guest WiFi.
    They want to know if one guest can see/attack another guest's traffic.

  Setup:
    • Connect Pi to hotel guest network via wlan0
    • Plug Alfa WiFi adapter for monitor mode on wlan1

  Steps:
    1. Voice: "network scan" — discover all guests on the subnet
    2. Note any hosts with open ports (SMB shares, Plex servers, etc.)
    3. Voice: "full auto" — AI tries ARP spoof, MitM positioning
    4. Check if the network segregates guest traffic (it should)
    5. Voice: "export" — full evidence report for hotel management

  Key findings this would surface:
    • Guests visible to each other (flat network, no client isolation)
    • Any unencrypted HTTP traffic between guests and internal services
    • Whether network allows scanning/probing between clients
    • Evidence of other guests with exposed Windows file shares

  ── SCENARIO 3: PRE-BREACH ASSESSMENT FOR RETAIL CHAIN ──────────

================================================================================
  SECTION 9 — NETWORK RECONNAISSANCE — TIPS & TRICKS
================================================================================

  QUICK COMMANDS (SSH into Pi and run manually for more control)
  ──────────────────────────────────────────────────────────────────────────────

  # Ping sweep — fastest host discovery
  nmap -sn 192.168.1.0/24

  # SYN scan all common ports (requires root)
  sudo nmap -sS -T4 --open 192.168.1.0/24

  # Full port scan on a specific host
  sudo nmap -sS -p- -T4 192.168.1.100

  # Service version detection
  sudo nmap -sV 192.168.1.100

  # OS fingerprinting
  sudo nmap -O 192.168.1.100

  # Aggressive scan (version + OS + scripts + traceroute)
  sudo nmap -A 192.168.1.100

  # Scan for specific vulnerabilities (NSE scripts)
  sudo nmap --script vuln 192.168.1.100

  # SMB vulnerabilities (EternalBlue check)
  sudo nmap --script smb-vuln-ms17-010 -p 445 192.168.1.0/24

  # HTTP title grabbing across whole subnet
  sudo nmap --script http-title -p 80,443,8080,8443 192.168.1.0/24

  # Scan with output to file
  sudo nmap -sS -oA /home/pi/reports/scan_result 192.168.1.0/24

  # ARP scan (faster than nmap ping on LAN)
  sudo arp-scan --localnet

  READING nmap OUTPUT
  ──────────────────────────────────────────────────────────────────────────────
  STATE    Meaning
  open     Port is accepting connections — service is running
  closed   Port is accessible but no service listening
  filtered Firewall or ACL is blocking the probe
  open|filtered  Cannot determine — common with UDP scans

  COMMON PORTS TO WATCH FOR
  ──────────────────────────────────────────────────────────────────────────────
  21   FTP       — often anonymous login or brute-forceable
  22   SSH       — brute force with hydra if password auth enabled
  23   Telnet    — unencrypted, try default credentials immediately
  25   SMTP      — open relay check, user enumeration
  80   HTTP      — full web pentest
  443  HTTPS     — web pentest + certificate check
  445  SMB       — EternalBlue, pass-the-hash, ransomware entry point
  3306 MySQL     — direct DB brute force if exposed
  3389 RDP       — Windows remote desktop, brute force
  5900 VNC       — often no authentication or weak password
  8080 HTTP-alt  — often admin panels, Jenkins, Tomcat
  8443 HTTPS-alt — alternative HTTPS admin panels

  MONITOR MODE SETUP (manual, for more control)
  ──────────────────────────────────────────────────────────────────────────────

  # Kill processes that interfere with monitor mode
  sudo airmon-ng check kill

  # Start monitor mode
  sudo airmon-ng start wlan1

  # Scan all channels for APs
  sudo airodump-ng wlan1mon

  # Lock onto specific BSSID and channel for handshake capture
  sudo airodump-ng --bssid AA:BB:CC:DD:EE:FF -c 6 -w /home/pi/reports/capture wlan1mon

  # Send deauth to force handshake (in a second terminal)
  sudo aireplay-ng -0 5 -a AA:BB:CC:DD:EE:FF wlan1mon

  # Crack the captured handshake
  aircrack-ng -w /home/pi/wordlists/rockyou.txt /home/pi/reports/capture-01.cap

  # Restore managed mode
  sudo airmon-ng stop wlan1mon
  sudo systemctl restart NetworkManager

  WORDLIST TIPS
  ──────────────────────────────────────────────────────────────────────────────
  • rockyou.txt (14 million passwords) — good for generic WiFi passwords
  • 10k-common.txt — fast first pass, catches most default passwords
  • Custom wordlist for target: combine company name + common suffixes
    Example: ClientName2024! ClientName123 ClientName@2024
  • Use crunch to generate custom wordlists:
    crunch 8 10 abcdefghijklmnopqrstuvwxyz0123456789 -o custom.txt

  WIFI SECURITY LEVELS (what to report to clients)
  ──────────────────────────────────────────────────────────────────────────────
  WEP         → CRITICAL — crackable in seconds regardless of password
  WPA2 + weak → HIGH     — crackable with rockyou in minutes
  WPA2 + strong → LOW    — dictionary attack fails, password is strong
  WPA3        → PASS     — PMKID attack not applicable, recommend maintaining
  Open (no password) → CRITICAL — immediate deauth and MitM possible

  HEADER ANALYSIS (what curl output tells you)
  ──────────────────────────────────────────────────────────────────────────────

  MISSING HEADERS = VULNERABILITIES TO REPORT:

  X-Frame-Options: DENY
    → If missing: clickjacking attack possible
    → Attacker can embed the site in an iframe and trick users

  Content-Security-Policy: default-src 'self'
    → If missing: XSS attacks are much easier to execute
    → No restriction on where scripts can be loaded from

  X-Content-Type-Options: nosniff
    → If missing: MIME-type sniffing attacks possible

  Strict-Transport-Security: max-age=31536000
    → If missing: SSL stripping attacks possible
    → Users connecting on HTTP can be downgraded

  Server: Apache/2.2.14
    → If present: version disclosure — attacker knows exact CVEs to use
    → Report as information disclosure, recommend hiding server header

  X-Powered-By: PHP/5.6.0
    → Same as above — technology stack disclosure
    → PHP 5.6 is end-of-life — critical finding

  Set-Cookie without Secure or HttpOnly flags
    → Session tokens can be stolen via XSS or HTTP interception

  COMMON WEB VULNERABILITIES TO INCLUDE IN REPORTS
  ──────────────────────────────────────────────────────────────────────────────
  • SQL Injection (SQLi)     — database compromise, data exfiltration
  • Cross-Site Scripting (XSS) — session hijacking, phishing
  • Insecure Direct Object Reference (IDOR) — access other users' data
  • Security Misconfiguration — exposed admin panels, default credentials
  • Sensitive Data Exposure — passwords in HTML source, API keys in JS
  • Broken Authentication — weak session tokens, no brute force protection
  • Directory Listing — web server lists all files in a directory

  # SSH brute force

  hydra -l administrator -P /home/pi/wordlists/10k-common.txt 192.168.1.10 rdp

  # MySQL brute force
  hydra -l root -P /home/pi/wordlists/10k-common.txt 192.168.1.10 mysql

  hydra -l admin -P /home/pi/wordlists/10k-common.txt 192.168.1.10 telnet

    192.168.1.10 ssh

  hydra -l admin -P wordlist.txt -t 4 192.168.1.10 ssh

  # Stop after first valid credential found
  hydra -l admin -P wordlist.txt -f 192.168.1.10 ssh

  SECTION 16 — TROUBLESHOOTING
================================================================================

  PROBLEM: OLED not showing anything
  ──────────────────────────────────────────────────────────────────────────────
  Check: i2cdetect -y 1
  Should show 0x3C. If blank: check I2C is enabled in raspi-config.
  Try: sudo raspi-config → Interface Options → I2C → Enable

  PROBLEM: "AI core unavailable — tool-only mode"
  ──────────────────────────────────────────────────────────────────────────────
  The LLM model file is missing or corrupt.
  Check: ls -lh /home/pi/models/tinyllama-1.1b-q4_0.gguf
  Should be ~620 MB. If missing, re-run: sudo ./setup.sh

  PROBLEM: "Voice input unavailable — Button C will act as SELECT"
  ──────────────────────────────────────────────────────────────────────────────
  Either vosk, pyaudio, or the Vosk model is missing.
  Check: ls /home/pi/models/vosk-model-small-en-us/
  If missing: wget https://alphacephei.com/vosk/models/vosk-model-small-en-us-0.15.zip
              unzip vosk-model-small-en-us-0.15.zip -d /home/pi/models/

  PROBLEM: WiFi crack fails at airmon-ng
  ──────────────────────────────────────────────────────────────────────────────
  Built-in wlan0 does not support monitor mode on Pi Zero 2 W.
  You MUST use a USB WiFi adapter with a monitor-mode capable chipset.
  Recommended: Alfa AWUS036ACH (rtl8812au chipset)
  After plugging in: change config.ini [wifi] interface = wlan1

  PROBLEM: nmap says "Failed to open device wlan0" or permission error
  ──────────────────────────────────────────────────────────────────────────────
  nmap SYN scan requires root. The systemd service runs as pi.
  Fix: sudo nmap ... or run the service as root:
  Edit /etc/systemd/system/pentestgpt.service → User=root
  Then: sudo systemctl daemon-reload && sudo systemctl restart pentestgpt

  PROBLEM: hydra says "Wordlist not found"
  ──────────────────────────────────────────────────────────────────────────────
  Check: ls -lh /home/pi/wordlists/
  If rockyou.txt missing: sudo apt install wordlists
  Then: gunzip -c /usr/share/wordlists/rockyou.txt.gz > /home/pi/wordlists/rockyou.txt

  PROBLEM: System runs out of memory during LLM load
  ──────────────────────────────────────────────────────────────────────────────
  Check: free -m   (should have at least 150 MB free before LLM loads)
  Increase swap: sudo dphys-swapfile swapoff
                 sudo sed -i 's/CONF_SWAPSIZE=.*/CONF_SWAPSIZE=256/' /etc/dphys-swapfile
                 sudo dphys-swapfile setup && sudo dphys-swapfile swapon

  PROBLEM: PiSugar 3 battery not detected (always shows 100%)
  ──────────────────────────────────────────────────────────────────────────────
  Check I2C: i2cdetect -y 1 (should show 0x57)
  Ensure PiSugar is properly seated on the Pi GPIO header.
  Check smbus2 is installed: pip show smbus2

  PROBLEM: espeak-ng not producing audio
  ──────────────────────────────────────────────────────────────────────────────
  Check speaker is connected to 3.5mm jack or USB.
  Test: espeak-ng "hello"
  Force audio to 3.5mm jack: sudo raspi-config → System Options → Audio → Headphones
  Check volume: alsamixer (set PCM to 100)

  PROBLEM: Service won't start — "Unit pentestgpt.service failed"
  ──────────────────────────────────────────────────────────────────────────────
  Check logs: sudo journalctl -u pentestgpt -n 50 --no-pager
  Most common causes:
    1. Python venv not found — re-run setup.sh
    2. main.py has a syntax error — run: python3 /home/pi/NxtGenAI/main.py
    3. Reports directory doesn't exist — run: mkdir -p /home/pi/reports
